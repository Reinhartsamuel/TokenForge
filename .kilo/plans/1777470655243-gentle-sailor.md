# Migration Plan: TokenForge → Canonical-First SSTS

## Context

TokenForge currently ships **its own custom SSTS program** (Anchor-based, 652 lines) in `programs/ssts/` alongside a custom FAMP compliance program (`programs/famp/`) and a no-op verification stub (`programs/verification_policy_noop/`). The canonical `solana-security-token-standard` repo (by Halborn/Hoodies/Solana Foundation) provides the authoritative SSTS implementation with 24 instructions, a transfer hook program, and a generated TypeScript client.

This migration replaces our custom SSTS program with the canonical dependency, reworks FAMP to plug into canonical SSTS as a verification program, and preserves the no-op stub for testing.

## What Exists vs. What Canonical Provides

| TokenForge Current | Canonical SSTS | Verdict |
|---|---|---|
| `programs/ssts/` (custom, Anchor) | `program/` (canonical, 24 instructions) | **DELETE our SSTS** — use canonical |
| `programs/famp/` (freeze/allowlist/blocklist) | Nothing equivalent (policy is external) | **KEEP but rework** — becomes a verification program |
| `programs/verification_policy_noop/` | `examples/verification-program-example/` | **KEEP** — still useful for testing |
| Custom Mint/Distribution/CorporateActions | `InitializeMint`, `CreateDistributionEscrow`, `Split`, `Convert` | **DELETE our implementations** — canonical covers all |
| Custom Freeze/Thaw/Pause/Resume | `Freeze`, `Thaw`, `Pause`, `Resume` | **DELETE our implementations** — canonical covers all |
| Custom verification CPI | `VerificationConfig` + `Verify` instruction | **DELETE our CPI** — canonical has its own verification model |
| Custom Merkle distribution | `CreateDistributionEscrow`, `ClaimDistribution`, `Proof` accounts | **DELETE our implementation** — canonical covers this |

## FAMP Decision: Keep or Remove?

**Decision: KEEP FAMP, rework as a canonical-compatible verification program.**

Rationale:
- Canonical SSTS has **no built-in allowlist/blocklist policy engine**. It delegates policy to external verification programs.
- FAMP's allowlist/blocklist + auto-freeze-on-removal functionality is **genuinely useful compliance logic** that the canonical framework expects you to implement yourself.
- FAMP becomes a **verification program** registered in `VerificationConfig` alongside (or instead of) the `transfer-whitelist` from `ssts-example`.
- FAMP's freeze/thaw functionality can be achieved via CPI to canonical SSTS's `Freeze`/`Thaw` instructions, but FAMP provides the policy logic for **when** to freeze.

**However**, FAMP currently uses Anchor and CPIs into Token-2022 directly. After migration, FAMP should:
1. Implement the `verify_transfer` interface expected by canonical SSTS (discriminator-based, positional accounts)
2. Use canonical SSTS's verification config system (be registered via `InitializeVerificationConfig`)
3. Still provide policy management instructions (`create_policy`, `add_to_allowlist`, etc.) as a separate interface
4. Auto-freeze via CPI into canonical SSTS (not directly into Token-2022)

**Alternative**: If FAMP's functionality is fully covered by combining canonical SSTS native Freeze/Thaw + the `transfer-whitelist` from `ssts-example`, then FAMP could be marked deprecated. But FAMP's blocklist + auto-freeze semantics are more comprehensive than a simple whitelist, so we keep it.

### Phase 3: Rework FAMP as Canonical-Compatible Verification Program

FAMP is reworked to function as a verification program within canonical SSTS's architecture, using the **Policy-Oracle Pattern** for freeze authority.

#### FAMP Authority Model
- **Issuer** is set as the Token-2022 freeze authority during `InitializeMint`
- FAMP acts as a **policy oracle**: it decides who should be frozen via `remove_from_allowlist` / `add_to_blocklist`
- The SDK/dashboard listens to FAMP events and constructs canonical SSTS `Freeze`/`Thaw` instructions signed by the issuer
- FAMP does NOT CPI into SSTS or Token-2022 for freeze/thaw — it only manages policy state

#### Implementation Tasks
- [ ] **Update FAMP's `verify_transfer` interface** to match canonical SSTS expectations:
  - Canonical verification programs receive the instruction discriminator + instruction data
  - Accounts are passed positionally — FAMP must handle both introspection and CPI account layouts
  - Return `Ok(())` to approve, error to reject
- [ ] **Add policy verification logic** to `verify_transfer`:
  - Look up `PolicyAccount` for the mint
  - Check allowlist: verify sender/receiver against `allowlist_merkle_root`
  - Check blocklist: verify sender/receiver against `blocklist_merkle_root`
  - Approve if allowlisted and not blocklisted
- [ ] **Remove FAMP's direct freeze/thaw CPI into Token-2022** — replaced by policy-oracle pattern
- [ ] **Keep FAMP's policy management instructions** unchanged:
  - `create_policy`, `add_to_allowlist`, `remove_from_allowlist`, `add_to_blocklist`, `remove_from_blocklist`
  - `thaw_token_account` — removed (issuer uses canonical SSTS `Thaw` directly)
- [ ] **Add event emission** to `remove_from_allowlist` and `add_to_blocklist`:
  - `WalletBlocked(wallet)` — signals the SDK to call SSTS `Freeze`
  - `WalletUnblocked(wallet)` — signals the SDK to call SSTS `Thaw` (if applicable)
- [ ] **Update FAMP tests** to use canonical SSTS programs in the test environment
- [ ] **Update FAMP Anchor.toml** to include canonical SSTS as a genesis/local dependency for testing

### Phase 4: Update Verification Policy No-Op

- [ ] Update `verification_policy_noop` to match canonical SSTS verification interface:
  - Must handle the same discriminator and account layout that canonical SSTS expects
  - Keep the always-approve behavior
- [ ] Update test configuration to reference canonical SSTS program IDs
- [ ] Keep as the default test verifier for TokenForge SDK development

### Phase 5: Build TokenForge SDK on Canonical Client

The SDK layer is TokenForge's main product. It must wrap the canonical TypeScript client:

- [ ] **L0**: Import canonical generated client from `solana-security-token-standard/clients/typescript`
  - Direct instruction builders and account types
  - No modifications to canonical code
- [ ] **L1** (TokenForge adapters): Add ergonomic wrappers:
  - Account derivation helpers (PDA seeds for VerificationConfig, Rate, Proof, Receipt, etc.)
  - Transaction composition utilities (e.g., "mint with verification" = verify + mint in one tx)
  - FAMP policy management wrappers
  - Error context enrichment
- [ ] **L2** (workflow APIs): High-level issuer operations:
  - `createSecurityToken(metadata, decimals, verificationProgram?)` → InitializeMint + InitializeVerificationConfig
  - `configureVerification(mint, instruction, programs, cpiMode)` → InitializeVerificationConfig / UpdateVerificationConfig
  - `mintTokens(mint, amount, destination, verificationContext?)` → Verify + Mint
  - `transferTokens(mint, amount, from, to, verificationContext?)` → Transfer (with verification)
  - `pauseToken(mint)` / `resumeToken(mint)`
  - `freezeAccount(mint, tokenAccount)` / `thawAccount(mint, tokenAccount)`
  - `createDistribution(mint, merkleRoot, amount, escrowToken)` → CreateDistributionEscrow
  - `claimDistribution(mint, claimArgs, proof?)` → ClaimDistribution
  - `executeSplit(mint, rateAccount, actionId)` → Split
  - `executeConvert(mint, rateAccount, amountToConvert)` → Convert
  - `can_add_to_allowlist(mint, wallet)`, `can_remove_from_allowlist(mint, wallet)`, etc.
- [ ] **Compatibility matrix**: SDK version → canonical SSTS release mapping
- [ ] **Tests**: Integration tests using canonical SSTS programs on local validator

## Risk Assessment & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Canonical SSTS program ID changes between devnet/mainnet deployment | High | Pin commit/tag; track deployments; update config per environment |
| FAMP CPI into SSTS Freeze/Thaw fails due to authority constraints | Medium | FAMP needs appropriate authority setup; use PermanentDelegate or configure as freeze authority |
| Canonical IDL changes break SDK wrappers | Medium | Version pinning + compatibility gate in CI; mark workflows unsupported if broken |
| Transfer hook program requires specific account layout | Medium | Study canonical transfer_hook source; test both introspection and CPI paths |
| Existing TokenForge users need migration path | High | Document breaking changes; provide migration guide; keep FAMP program ID stable |

## Resolved Decisions

1. **FAMP authority model**: Policy-Oracle Pattern — FAMP emits events, SDK executes canonical SSTS `Freeze`/`Thaw`
2. **Canonical integration**: Git submodule at `lib/canonical-ssts`
3. **FAMP verification scope**: Transfers only (discriminator 12)
4. **Verification program offering**: FAMP only — no `transfer-whitelist` from `ssts-example`
