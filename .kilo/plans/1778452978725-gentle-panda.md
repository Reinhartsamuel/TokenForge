# TokenForge SaaS Dashboard Transformation

## Overview

Transform the current single-page form-based dashboard into a full SaaS-style issuer management platform with PostgreSQL persistence, activity tracking, and professional data-dense UI.

**Current State**: Single page with 5 tab buttons → forms → console-like result text
**Target State**: Multi-page SaaS dashboard with sidebar navigation, data tables, history tracking, and operational insights

---

## Architecture

### Technology Stack
- **Database**: PostgreSQL 16+ (Railway)
- **ORM**: Drizzle ORM (lightweight, TypeScript-native, works well with Next.js)
- **Connection Pooling**: Railway PostgreSQL direct connection
- **Existing**: Next.js 16.2.4, React 19, shadcn/ui, Tailwind CSS v4, @solana/kit v2.1.0

### Design System (from ui-ux-pro-max)
- **Pattern**: Data-Dense + Drill-Down
- **Colors**: Primary `#0369A1`, Secondary `#0EA5E9`, CTA `#22C55E`, Background `#F0F9FF`, Text `#0C4A6E`
- **Typography**: Fira Code (monospace for addresses/hashes) / Fira Sans (body/UI)
- **Effects**: Hover tooltips, row highlighting, smooth filter animations, data loading spinners
- **Style**: Data-Dense Dashboard — space-efficient, maximum data visibility

## Known Issues & Fixes

### Drizzle ORM Query Builder Type Mismatch
**File**: `dashboard/app/api/tokens/route.ts`
**Error**: Property 'where' is missing in type when reassigning query variable
**Fix**: Restructure query to use conditional branching instead of reassigning:
```typescript
let result;
if (search) {
  result = await db.select().from(tokens).where(or(...)).orderBy(...).limit(...).offset(...);
} else {
  result = await db.select().from(tokens).orderBy(...).limit(...).offset(...);
}
```

### DropdownMenuTrigger asChild Issue
**Files**: `dashboard/app/activity/page.tsx`, `dashboard/app/tokens/page.tsx`
**Error**: Type '{ children: Element; asChild: true; }' is not assignable to type 'IntrinsicAttributes & Props<unknown>'
**Fix**: Remove `asChild` prop since `@base-ui/react` dropdown components don't support it:
```typescript
<DropdownMenuTrigger>  // Remove asChild
  <Button variant="outline" size="sm">
```

### 1.3 Database Schema (Drizzle)

File: `dashboard/db/schema.ts`

```typescript
// tokens table
tokens: {
  id: uuid PK (default gen_random_uuid())
  mintAddress: varchar(44) UNIQUE NOT NULL
  name: varchar(255) NOT NULL
  symbol: varchar(10) NOT NULL
  decimals: integer NOT NULL DEFAULT 6
  uri: text  -- R2 metadata URL
  description: text
  imageUrl: text
  mintAuthorityPda: varchar(44)
  freezeAuthorityPda: varchar(44)
  verificationConfigPda: varchar(44)
  creatorAddress: varchar(44) NOT NULL
  network: varchar(10) NOT NULL DEFAULT 'devnet'  -- devnet | mainnet
  status: varchar(20) NOT NULL DEFAULT 'active'   -- active | paused | frozen
  createdAt: timestamp DEFAULT now()
  updatedAt: timestamp DEFAULT now()
}

// transactions table (activity log)
transactions: {
  id: uuid PK
  tokenId: uuid FK -> tokens.id
  signature: varchar(88) NOT NULL  -- Solana tx signature
  type: varchar(50) NOT NULL  -- create | mint | transfer | policy_create | policy_allowlist | policy_blocklist | distribution_create | distribution_claim | pause | resume | freeze | thaw
  status: varchar(20) NOT NULL  -- pending | confirmed | failed
  fromAddress: varchar(44)  -- sender or issuer
  toAddress: varchar(44)  -- receiver or target
  amount: numeric(30, 9)  -- token amount (nullable for non-mint/transfer)
  metadata: jsonb  -- additional context (PDA addresses, policy details, etc.)
  explorerUrl: text
  createdAt: timestamp DEFAULT now()
}

// famp_policies table
famp_policies: {
  id: uuid PK
  tokenId: uuid UNIQUE FK -> tokens.id
  policyAddress: varchar(44) UNIQUE
  allowlistMode: boolean DEFAULT false
  createdAt: timestamp DEFAULT now()
  updatedAt: timestamp DEFAULT now()
}

// famp_policy_entries table (allowlist/blocklist)
famp_policy_entries: {
  id: uuid PK
  policyId: uuid FK -> famp_policies.id
  walletAddress: varchar(44) NOT NULL
  listType: varchar(20) NOT NULL  -- allowlist | blocklist
  createdAt: timestamp DEFAULT now()
  unique(walletAddress, policyId, listType)
}

// distributions table
distributions: {
  id: uuid PK
  tokenId: uuid FK -> tokens.id
  escrowAta: varchar(44)
  merkleRoot: varchar(66) NOT NULL
  actionId: integer NOT NULL DEFAULT 0
  status: varchar(20) NOT NULL DEFAULT 'active'  -- active | completed | cancelled
  totalClaimed: numeric(30, 9) DEFAULT 0
  totalClaimants: integer DEFAULT 0
  createdAt: timestamp DEFAULT now()
  updatedAt: timestamp DEFAULT now()
}

// distribution_claims table
distribution_claims: {
  id: uuid PK
  distributionId: uuid FK -> distributions.id
  claimantAddress: varchar(44) NOT NULL
  amount: numeric(30, 9) NOT NULL
  leafIndex: integer
  status: varchar(20) NOT NULL DEFAULT 'pending'  -- pending | claimed | failed
  signature: varchar(88)  -- claim tx signature
  claimedAt: timestamp
  createdAt: timestamp DEFAULT now()
}
```

### 1.4 Drizzle Config
File: `dashboard/drizzle.config.ts`
```typescript
export default {
  schema: "./db/schema.ts",
  out: "./db/migrations",
  dialect: "postgresql",
  dbCredentials: { url: process.env.DATABASE_URL! },
};
```

### 1.5 Database Client
File: `dashboard/lib/db.ts`
```typescript
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "@/db/schema";

const client = postgres(process.env.DATABASE_URL!);
export const db = drizzle(client, { schema });
```

Add to `dashboard/package.json`:
```json

"db:studio": "drizzle-kit studio"
```

## Phase 2: API Routes for Database CRUD

Keep existing Solana transaction API routes. Add new routes for data management:

| Route | Method | Purpose |
|-------|--------|---------|
| `/api/tokens` | GET | List all tokens with pagination |
| `/api/tokens` | POST | Create token record (called after on-chain creation) |
| `/api/tokens/[mintAddress]` | GET | Get single token details |
| `/api/tokens/[mintAddress]` | PATCH | Update token status (pause/resume) |
| `/api/transactions` | GET | List transactions with filters (token, type, date range) |
| `/api/transactions` | POST | Record transaction after on-chain execution |
| `/api/famp/[mintAddress]` | GET | Get FAMP policy with entries |
| `/api/famp/[mintAddress]/entries` | POST | Add/remove policy entries |
| `/api/distributions` | GET | List all distributions |
| `/api/distributions` | POST | Create distribution record |
| `/api/distributions/[id]` | GET | Get distribution with claims |
| `/api/distributions/[id]/claims` | POST | Record claim |
| `/api/stats` | GET | Dashboard overview metrics |

**Key Pattern**: After each on-chain transaction succeeds, the API route should also write the corresponding database record.

## Phase 5: Integration Points

### 5.1 Transaction Recording Middleware

After each existing API route succeeds (token creation, mint, transfer, FAMP, distribution), write to the database:

```typescript
// Example: In /api/test-create-token/route.ts, after on-chain success:
await db.insert(transactions).values({
  tokenId: token.id,
  signature: data.signature,
  type: "create",
  status: "confirmed",
  fromAddress: creatorAddress,
  metadata: {
    mintAuthorityPda: data.mintAuthorityPda,
    verificationConfigPda: data.verificationConfigPda,
  },
  explorerUrl: data.explorerUrl,
});
```

### 5.2 Token Creation Flow
1. User fills form → clicks "Create"
2. On-chain transaction executes (existing logic)
3. On success: Insert into `tokens` table
4. Insert into `transactions` table
5. Redirect to token detail page

### 5.3 FAMP Policy Flow
1. Load policy from on-chain (existing)
2. Also load from `famp_policies` and `famp_policy_entries` tables
3. On successful on-chain policy action: update DB
4. DB serves as fast-read cache; on-chain is source of truth

## Tradeoffs & Decisions

### Q: Why Drizzle over Prisma?
- Drizzle is lighter, faster cold starts (important for serverless/Edge)
- Better TypeScript inference without codegen step
- SQL-like API familiar to developers

### Q: Database hosting
- Using Railway PostgreSQL for managed hosting
- Connection string will be stored in `.env` as `DATABASE_URL`
- Railway provides standard PostgreSQL 16+ with connection pooling
